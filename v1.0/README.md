# Printer Shop MongoDB Stand

Учебный стенд для практической работы по дисциплине «Администрирование баз данных». Проект моделирует интернет-магазин принтеров на MongoDB и показывает полный цикл разработки стенда: инициализацию БД, валидацию схем, роли доступа, демонстрационные данные, API на FastAPI и веб-интерфейс для проверки сценариев.

## Дополнительный файл по запросам

В корне проекта есть [`requests.md`](requests.md) - короткая сводка по 10 запросам из задания. В нём собраны:

- какой запрос где реализован в интерфейсе
- кто может его открыть по роли
- что именно нужно нажать или выбрать

Это удобно использовать как шпаргалку при проверке сценариев в браузере.

## Что входит в проект

- MongoDB в Docker с автоматической инициализацией
- `mongo-express` для просмотра данных
- FastAPI API
- веб-интерфейс для каталога, корзины, заказов, администрирования клиентов, управления товарами и аналитики
- отдельный seed-скрипт для повторного заполнения БД без пересоздания Docker volume

## Требования

- Linux
- Docker и Docker Compose Plugin
- Python 3.11+ или 3.12
- `pip`

## Подготовка окружения

Создать виртуальное окружение и установить зависимости:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Создать `.env`:

```bash
cp .env.example .env
```

Основное подключение приложения к БД:

```env
MONGO_URI=mongodb://admin:admin123@localhost:27017/shop?authSource=admin
```

## Запуск MongoDB

На чистом volume контейнер MongoDB автоматически выполнит `docker/mongo-init/01-init-shop.js` и создаст:

- базу данных `shop`
- коллекции `categories`, `products`, `customers`, `employees`, `orders`, `carts`
- `$jsonSchema` валидаторы
- индексы
- демонстрационные данные
- MongoDB roles и users для практической работы

Запуск:

```bash
docker compose up -d
docker compose ps
```

Если нужна гарантированно чистая инициализация:

```bash
docker compose down -v
docker compose up -d
```

## Повторное заполнение БД без удаления volume

Если контейнер уже работает и нужно заново загрузить демоданные:

```bash
cat docker/seed-from-friend-compose.js | docker exec -i printer_store_mongo mongosh --username admin --password admin123 --authenticationDatabase admin
```

## Запуск FastAPI

```bash
source .venv/bin/activate
python -m uvicorn app.main:app --reload
```

Полезные адреса:

- UI: `http://127.0.0.1:8000/`
- Swagger: `http://127.0.0.1:8000/docs`
- Health: `http://127.0.0.1:8000/health`
- mongo-express: `http://127.0.0.1:8081`

Ожидаемый ответ health-check:

```json
{"status":"ok","database":"connected"}
```

## Подробное описание базы данных и функций администрирования

Этот раздел является ключевым для практической работы, потому что основная цель стенда заключается не только в создании веб-интерфейса, а прежде всего в проектировании, эксплуатации и администрировании MongoDB как рабочей базы данных интернет-магазина.

### Назначение базы данных `shop`

База данных `shop` моделирует предметную область интернет-магазина принтеров. Она хранит:

- справочник категорий товаров
- каталог товаров с характеристиками
- данные покупателей
- данные сотрудников
- корзины покупателей
- оформленные заказы

Таким образом, база покрывает как операционные сценарии магазина, так и административные задачи:

- создание структуры хранения данных
- загрузку демонстрационного наполнения
- контроль корректности документов
- разграничение прав доступа
- ускорение запросов за счёт индексов
- выполнение прикладной аналитики

### Состав коллекций

В `shop` используются следующие коллекции:

- `categories` — категории товаров
- `products` — каталог товаров
- `customers` — покупатели
- `employees` — сотрудники магазина
- `orders` — оформленные заказы
- `carts` — корзины покупателей

Каждая из коллекций отвечает за отдельную сущность предметной области и используется приложением FastAPI напрямую.

### Логическая модель данных

#### `categories`

Коллекция категорий хранит минимальный справочник:

- `_id`
- `name`

Назначение:

- классификация товаров
- фильтрация каталога
- участие в аналитических запросах по спросу на категории

#### `products`

Коллекция товаров является центральной частью базы. Документ товара содержит:

- `_id`
- `name`
- `manufacturer`
- `technology`
- `paperFormat`
- `colors`
- `price`
- `quantity`
- `categoryId`
- `characteristics`

Назначение полей:

- `name` — название товара
- `manufacturer` — производитель
- `technology` — технология печати
- `paperFormat` — поддерживаемый формат бумаги
- `colors` — количество цветов
- `price` — цена
- `quantity` — остаток на складе
- `categoryId` — ссылка на категорию
- `characteristics` — дополнительные характеристики, например скорость печати, разрешение и ёмкость лотка

Эта коллекция демонстрирует:

- проектирование структуры документов
- хранение смешанных данных: обязательных и расширяемых характеристик
- фильтрацию по нескольким атрибутам
- текстовый поиск

#### `customers`

Коллекция клиентов хранит:

- `_id`
- `firstName`
- `lastName`
- `email`
- `phone`

Назначение:

- идентификация покупателя
- связь с корзиной
- связь с заказами
- демонстрация администрирования клиентских данных

#### `employees`

Коллекция сотрудников хранит:

- `_id`
- `firstName`
- `lastName`
- `position`
- `email`
- `salary`

Она нужна для:

- моделирования ответственного сотрудника по заказу
- демонстрации предметной области магазина
- использования референсов в заказах

#### `carts`

Коллекция корзин содержит:

- `customerId`
- `products`

Каждый элемент массива `products` хранит:

- `productId`
- `name`
- `quantity`
- `price`

Назначение:

- промежуточное хранение выбранных товаров до оформления заказа
- расчёт итоговой суммы
- демонстрация работы с вложенными массивами документов

#### `orders`

Коллекция заказов содержит:

- `_id`
- `customerId`
- `employeeId`
- `products`
- `totalPrice`
- `status`
- `deliveryAddress`
- `orderDate`

Каждый заказ хранит срез состояния корзины на момент оформления. Это важно с точки зрения БД, потому что заказ должен быть независим от последующих изменений в карточке товара.

Коллекция используется для:

- просмотра заказов клиента
- изменения статусов менеджером
- аналитики продаж

### Валидация данных через `$jsonSchema`

Одной из важнейших частей работы является не просто хранение документов, а контроль их корректности на уровне самой MongoDB.

В проекте для коллекций заданы `$jsonSchema` валидаторы с режимами:

- `validationLevel: "strict"`
- `validationAction: "error"`

Это означает:

- MongoDB строго проверяет структуру документа
- некорректные документы не попадают в базу
- приложение не является единственной линией защиты

Примеры того, что контролируется валидаторами:

- обязательные поля товара
- допустимые значения `technology`
- допустимые значения `paperFormat`
- типы числовых полей
- обязательные поля клиента
- обязательные поля заказа
- структура вложенных элементов в `orders.products` и `carts.products`

С точки зрения дисциплины это важно, потому что показывает именно администрирование БД, а не только программную валидацию в коде.

### Индексы и оптимизация запросов

Для учебного стенда настроены индексы, которые отражают реальные задачи ускорения доступа к данным:

- текстовый индекс по `products.name` и `products.manufacturer`
- индекс `products.categoryId`
- составной индекс `products.technology + products.paperFormat`
- индекс `orders.customerId + orders.orderDate`
- индекс `orders.status`
- уникальный индекс `carts.customerId`
- уникальный индекс `customers.email`

Назначение индексов:

- ускорение поиска по каталогу
- ускорение фильтрации по категории и характеристикам
- ускорение получения заказов клиента
- ускорение поиска заказов по статусу
- обеспечение уникальности корзины для клиента
- защита от дублирования email у клиентов

Это позволяет показать, что база данных не просто хранит документы, а администрируется с учётом производительности и ограничений целостности.

### Роли и пользователи MongoDB

В стенде присутствуют не только прикладные роли интерфейса, но и реальные пользователи MongoDB.

Создаются следующие роли:

- `shopAdminRole`
- `shopManagerRole`
- `shopUserRole`
- `shopGuestRole`

Создаются следующие пользователи:

- `shop_admin`
- `shop_manager`
- `shop_user`
- `shop_guest`

Их назначение:

- `shop_admin` — полный доступ к базе `shop`
- `shop_manager` — работа с товарами, категориями и заказами
- `shop_user` — чтение каталога и работа с корзиной/заказами
- `shop_guest` — только чтение каталога и категорий

Это особенно важно для практической работы, потому что показывает настоящее разграничение доступа внутри СУБД, а не только проверку прав внутри Python-кода.

### Сценарии администрирования, которые демонстрирует стенд

Стенд позволяет показать следующие административные задачи:

- автоматическое развёртывание структуры БД
- создание коллекций с валидаторами
- создание индексов
- создание ролей и пользователей MongoDB
- загрузку тестовых данных
- повторное развёртывание БД на чистом volume
- повторное наполнение БД seed-скриптом
- проверку ограничений через попытку вставки некорректных документов
- проверку прав доступа разными пользователями MongoDB

Таким образом, проект соответствует логике дисциплины: мы не просто “используем MongoDB”, а управляем её структурой, наполнением, доступом и качеством данных.

### Как база данных связана с приложением

FastAPI в этом проекте служит прикладным слоем над MongoDB. Через него демонстрируются реальные операции над данными:

- получение категорий
- фильтрация товаров
- добавление товаров в корзину
- оформление заказа
- получение списка заказов клиента
- изменение статуса заказа
- управление клиентами администратором
- выполнение аналитических запросов

Но источником истины остаётся именно MongoDB:

- структура коллекций задаётся init-скриптом
- корректность документов контролирует MongoDB
- роли и пользователи создаются в MongoDB
- аналитика выполняется на основании данных MongoDB

### Аналитические возможности базы данных

Важной частью стенда являются аналитические запросы, потому что они показывают использование БД не только в транзакционном режиме, но и как инструмента анализа данных.

Поддерживаются следующие аналитические сценарии:

- топ продаж за период
- клиенты с количеством заказов больше заданного
- спрос по категориям за период
- товары без продаж на выбранную дату

Эти запросы используют агрегирование MongoDB и демонстрируют:

- работу с массивами вложенных документов
- группировку данных
- вычисление суммарных показателей
- связывание данных между коллекциями

Такой формат делает историю версий полезной не только как историю кода, но и как историю развития архитектуры базы данных, что особенно важно для дисциплины «Администрирование баз данных».
